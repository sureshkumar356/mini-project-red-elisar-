SLIDE 6: RAG Pipeline — 4 Phases in Detail
--------------------------------------------------
HEADING: How RAG Works — Our 4-Phase Pipeline

DIAGRAM: Vertical or circular flow showing 4 phases:

  PHASE 1: RETRIEVAL
    Input: Attack scenario text
    Process: Sentence Transformer → 384-dim vector → FAISS search
    Output: Top-5 relevant MITRE techniques with similarity scores
    Time: ~45ms

  PHASE 2: PROMPT AUGMENTATION
    Input: Top-5 retrieved techniques + original scenario
    Process: Format techniques as structured context cards
    Output: Augmented system prompt (~3000-5000 characters)
    Key Feature: Explicit instruction "Use ONLY retrieved techniques"
    Time: ~3ms

  PHASE 3: LLM GENERATION
    Input: Augmented system + user prompts
    Process: Mistral 7B (via Ollama) generates JSON attack chain
    Output: N-step attack chain in JSON format
    Settings: temp=0.2, top_p=0.9, repeat_penalty=1.1
    Time: ~28 seconds (CPU inference)

  PHASE 4: VALIDATION
    Input: Raw LLM JSON response
    Process: JSON parse → Schema validate → Faithfulness check
    Output: Validated chain + faithfulness score + warnings
    Anti-hallucination: Cross-check all technique IDs vs retrieved set
    Time: ~1ms

SLIDE 8: Vector Store — FAISS HNSW Indexing
--------------------------------------------------
HEADING: Efficient Semantic Search with FAISS HNSW Vector Index

CONTENT:

  WHY FAISS?
    - 700+ techniques: too many for sequential comparison
    - Need: millisecond-speed similarity search at scale
    - Solution: FAISS HNSW (Hierarchical Navigable Small World) index

  HNSW ARCHITECTURE:
    - Graph-based Approximate Nearest Neighbor (ANN) search
    - Multi-layer graph: top layers (coarse), bottom layers (fine)
    - Each vector connected to M=48 neighbors
    - Search explores efSearch=32 candidates per layer

  CONFIGURATION:
    Parameter            Value    Description
    ----------------     -----   ---------------------------------
    Embedding Model      MiniLM  all-MiniLM-L6-v2
    Vector Dimension     384     384 float32 numbers per technique
    Index Type           HNSW    Hierarchical Navigable Small World
    M (connections)      48      Neighbors per node
    efSearch             32      Candidates explored per search
    efConstruction       200     Candidates during index build
    Similarity Metric    Cosine  Via L2-normalized vectors
    Top-K Retrieved      5       Most relevant techniques per query
    Retrieval Latency    ~45ms   On a 16GB laptop (CPU only)

  DIVERSE RETRIEVAL:
    - Retrieves top-15 then selects 5 across multiple kill chain tactics
    - Ensures attack chains don't cluster in one phase
    - Key tactics prioritized: initial-access, execution, persistence,
      privilege-escalation, lateral-movement, exfiltration

SLIDE 9: Anti-Hallucination System
--------------------------------------------------
HEADING: Grounded Generation — The Anti-Hallucination Framework

CONTENT:

  THE HALLUCINATION PROBLEM:
    When LLMs are asked about MITRE ATT&CK techniques, they can:
    - Invent technique IDs that don't exist (e.g., "T1234 Fake Attack")
    - Mix up technique descriptions
    - Reference old, deprecated techniques

  OUR 3-LAYER SOLUTION:

    LAYER 1 — CONSTRAINED PROMPTING:
      System prompt explicitly states:
      "You MUST ONLY use techniques from the RETRIEVED CONTEXT below.
       Do NOT invent or reference any technique not present in the context.
       Every technique you cite MUST include its exact Technique ID from the context."

    LAYER 2 — POST-GENERATION VERIFICATION:
      After the LLM generates:
      ✓ Extract all technique IDs from the generated chain
      ✓ Check each ID against the known retrieved set
      ✗ Flag any ID not found → _hallucination_flag: True
      ✓ Log a warning for each hallucinated reference

    LAYER 3 — FAITHFULNESS SCORING:
      Faithfulness = Grounded Steps / Total Steps
      With RAG:    Faithfulness ≈ 91% (Review 2 result)
      Without RAG: Faithfulness ≈ 45% (baseline)

  EXAMPLE:
    Retrieved techniques: [T1566.001, T1059, T1053, T1134, T1041]
    LLM generates chain with: [T1566.001, T1059, T1053, T9999, T1041]
    T9999 does not exist → flagged as hallucinated → warning logged
    Faithfulness = 4/5 = 80%

INTRODUCTION (Slide 1-2):

ARCHITECTURE (Slide 5):
  "The system works in four phases. First, we retrieve relevant MITRE ATT&CK
   techniques from our FAISS vector index using semantic search. Then we augment
   the LLM's prompt with these retrieved techniques. Third, the local Mistral model
   generates the attack chain. Finally, we validate every technique ID against the
   retrieved set to catch any hallucinations."

ANTI-HALLUCINATION (Slide 9):
  "This is one of our key differentiators. Plain LLMs hallucinate — they invent
  technique IDs that don't exist. Our system reduces this by constraining the
  LLM to retrieved MITRE context and then verifying technique IDs after
  generation. In evaluation, this grounding shows up as non-zero Context
  Relevance for RAG (vs 0.00 for No-RAG baselines)."

RESULTS (Slides 16-17):
  "In our exported evaluation, Red ELISAR (RAG) reaches F1 = 0.8621 versus
   0.6647 (Mistral No-RAG) and 0.6371 (LLaMA 3 No-RAG). Context relevance is
   0.82 for RAG and 0.00 for No-RAG baselines, showing the generator is grounded
   in retrieved MITRE context. Latency remains ~480ms with minimal overhead."

CONCLUSION (Slide 20):
  "We have demonstrated that a privacy-preserving, local-only RAG architecture
   can generate high-quality, faithful attack chains from MITRE ATT&CK data.
   The system runs entirely on a 16GB laptop, making professional red team
   simulation accessible without cloud infrastructure."
